# Codex CLI Execution Contract

## Purpose

This specification defines the MVP contract between the Rust backend and Codex CLI so agent execution can be implemented and tested without guessing the process shape.

## Execution Model

The backend launches Codex CLI as a child process for authoring jobs only.

Each execution must:

- run with the conversation-owned book workspace as the working directory
- receive a single prompt package file path or inline prompt payload generated by the backend
- run under a per-job timeout
- capture exit status, stdout, and stderr

Control commands must never invoke Codex CLI.

## MVP Invocation Shape

The backend implementation should isolate the process launcher behind an interface, but the contract for the real CLI path should be:

- executable: configured by environment, for example `CODEX_CLI_PATH`
- arguments: configured by backend code and documented in one place
- stdin: optional
- workspace cwd: required

The backend must support a fake executor in tests that implements the same interface and returns deterministic outputs.

## MVP Decisions In This Repository

- The backend passes the prompt package as stdin text rather than by temporary file path.
- The backend executes authoring jobs synchronously for MVP.
- The backend serializes authoring execution per conversation so two jobs for the same conversation cannot run at the same time.
- On timeout, the backend terminates the Codex CLI child process and records the run as `timed_out`.

## Prompt Package Contract

The backend must construct one prompt package per authoring job containing:

- normalized user instruction
- system constraints
- current book metadata summary
- current manuscript structure summary
- recent conversation summary when available
- explicit file-scope and output constraints

The prompt package may be passed either:

- as stdin text
- as a temporary prompt file path argument

For MVP, either approach is acceptable as long as it is documented in the implementation and exercised by tests.

## Success Contract

A successful execution is defined by all of the following:

- process exits with code `0`
- the workspace remains readable after execution
- the backend can determine the changed files set
- the backend can generate a revision summary and refresh render state

## Failure Contract

The backend must classify at least:

- non-zero exit as `failed`
- timeout as `timed_out`
- launcher/configuration failure as `failed`
- post-run render refresh failure as `failed` with a distinct internal reason

In this repository's MVP implementation, post-run render refresh failure also prevents revision creation for that run, so the latest successful revision remains the reader-visible revision.

## Changed Files Determination

The backend must determine changed files by taking a workspace snapshot before and after execution and computing the file delta within the workspace root.

The changed-files result must:

- be limited to files under the conversation workspace
- include created, modified, and deleted files
- never rely on Codex CLI to self-report the canonical change set

## Result Record

The executor interface must return or allow the backend to derive:

- process outcome classification
- exit code if available
- timed-out flag
- stdout summary
- stderr summary
- changed files

The backend then remains responsible for:

- writing job state transitions
- creating revisions on success only
- triggering render refresh
- producing user-facing notification text

## Testability Requirement

The execution layer must be implemented behind a trait or equivalent abstraction so integration tests can use a fake executor with scripted outcomes:

- success with file updates
- failure with stderr
- timeout without completion
